Text Description of the Diagram
This diagram shows how my load balancer simulation is organized. The LoadBalancer class is the main controller where incoming web requests are stored in one main queue, and then sent to a group of WebServer objects to be handled. Each Request includes a source IP, a destination IP, how many cycles it needs to finish, serviceTime, and a jobType letter that marks it as processing or streaming. The LoadBalancer owns the requestQueue, the list of servers, and the cooldownTimer. The simulation starts in run(), where it first creates the initial queue by calling generateInitialQueue() and also prepares the system to start running cycles. During each cycle, randomIncomingRequests() may randomly create and add new requests. Before a request is added into the queue, the IPBlocker checks whether the incoming IP is blocked. If it is blocked, the request is dropped and recorded as blocked in Stats, and if it is allowed, it is added to the queue and recorded as accepted. After that, the LoadBalancer calls oneCyclePast() to move each WebServer forward by one cycle. If a server is busy, it decreases remainingCycle. When the remainingCycle reaches zero, the server finishes the request, becomes available again, and Stats counts the request as completed. Next, dispatchRequests() assigns new work to any available server by popping from the queue and calling WebServer.assign(request), which sets the server's currentRequest and remainingCycle. To prevent the system from being overloaded or underused, scaleIfNeeded() can add or remove servers based on the current queue size. It also uses cooldownTimer so the number of servers does not change too often. Throughout the simulation, Logger.log and Logger.logCycleSummary record important events, and Stats keeps track of totals such as requests generated, accepted, blocked, and completed, how many times servers scaled up or down, and the maximum queue size seen. Finally, Config holds the system settings so the user can easily change values like the starting number of servers, total cycles, initial queue size, cooldown time, and request probabilities.
